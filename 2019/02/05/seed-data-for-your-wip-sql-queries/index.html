In my last post, I wrote about steps you can take to make writing complicated queries more manageable. One aspect that I didn’t cover in that post is how to set sample data to work with during the writing process. Assuming you’re not working directly in your production database as you test out new queries (right? right?? right???), you need some way to work on your new ideas.

In this post, I want to share some tips and tricks for creating reliable, reproduceable test data to help you develop new ideas in SQL.

Define Your Work Space

As much as I admire people who dive head-first into the command line for interacting with a database (and aspire to be that kind of person), when it comes to developing a query that I hope will someday live in production, every part of that process should really find a home as lines of text in your editor of choice.

Open a new SQL file and let’s get started.

First, even if you’re dealing with existing tables, let’s lay down the basic structure of your tables in DDL that we can copy and past for repeated use.

In the previous post, we were working with students and quizzes and scores, so the tables we needed ended up looking like this:

CREATE TABLE students (id serial primary key, first_name text, last_name text);
CREATE TABLE quizzes (id serial primary key, quiz_date date, topic text);
CREATE TABLE quiz_results (quiz_id int references quizzes(id), student_id int references students(id), score float);

If you’ve ever used a tool for managing database migrations, like ActiveRecord in Ruby or Alembic in Python, you’re already familiar with the concept of writing SQL that can go “up” or “down”. That is, your tables can be created and defined on the “up” migration and deleted on the “down” migration. We’re basically trying to do the same thing here.

There are two reasons this is an important habit:

  1. It’s pain in the ass to type this all out in the console from scratch each time I change my mind and need to start over.
  2. Ultimately, I want to share this with my coworkers or enshrine it forever in a code repository, so it’s convenient to have a record of exactly what I created when I ran the experiment.

Go read Caitlin Hudon’s excellent post about all the reasons you should capture your SQL in git. Even at this stage, you should start the process of building a valuable code artifact.

And if you want to take application development in SQL as far as humanly possible, I also recommend reading Mastering Postgres. You’ll come away thinking you can throw all your frameworks in the trash and be a SQL command line cowboy for everything (again, something I aspire to, but maybe I’m not ready to abandon everything but psql).

Back to developing our little SQL idea…once we have our tables, we need to add data.

No way! Enter Faker.

Faker is a Ruby gem that helps you generate placeholder data, like Lorem Ipsum for your database. Currently, there are over 100 different random data generators. I used scientists, rock bands, Seinfeld characters, and World Cup stadiums to seed data for the student quizzes and those aren’t even the weird ones.

To generate each of the insert statements above, I used a simple ruby script + Faker. I want the output to be a valid SQL statement that I can copy and paste into my growing seed file.

Here’s what the script looks like to generate our students:

require 'faker'

students = []
100.times do
  students << [Faker::Name.first_name, Faker::Name.last_name]
end

values = students.map{ |s| "(\"#{s.first}\",\"#{s.last}\")"}.join(',')
s = "INSERT INTO students (first_name, last_name) VALUES #{values};"

puts s

Repeat for quizzes:

quizzes = []
(0..90).to_a.reverse.each do |offset|
  quiz_date = "now() - interval '#{offset} days'"

  topic = if offset % 10 == 0
    Faker::Science.scientist
  elsif offset % 3 == 0
    Faker::RockBand.name
  elsif offset % 7 == 0
    Faker::WorldCup.stadium
  else
    Faker::Seinfeld.character
  end

  quizzes << [quiz_date, topic]
end

quiz_values = quizzes.map { |q| "(#{q.first}, '#{q.last}')"}.join(',')
s = "INSERT INTO quizzes (quiz_date, topic) VALUES #{quiz_values};"

puts s

And quiz scores:

quiz_results = []
(93..183).to_a.each do |quiz_id|
  (2..101).to_a.each do |student_id|
    base = (72..99).to_a
    incr = (0..9).to_a
    score = base.sample + 0.1 * incr.sample
    quiz_results << [quiz_id, student_id, score]
  end
end

quiz_result_values = quiz_results.map { |q| "(#{q[0]}, #{q[1]}, #{q[2]})"}.join(',')
s = "INSERT INTO quiz_results (quiz_id, student_id, score) VALUES #{quiz_result_values};"

puts s

The idea is to print out all the SQL strings, paste them into the seed file, and then copy/paste again into the SQL console to actually fill up the tables.

At any point in developing the example, if I decide that the data isn’t quite what I need, I can painlessly tear down the tables, tweak the table definitions or the ruby script, then just replay everything. Really easy!

In fact, as I was going along writing the post, I decided the data would be more interesting if there was a little less of it. I opened the seed file, wrote a couple of delete statements, and when I liked the way the data looked, saved them as part of the seed file:

DELETE FROM quiz_results WHERE student_id > 80 and quiz_id IN (SELECT id FROM quizzes WHERE quiz_date < '2018-12-01');

DELETE FROM quiz_results WHERE student_id > 50 and quiz_id IN (SELECT id FROM quizzes WHERE quiz_date < '2018-11-01');

Hopefully, this all looks like a simple process. It can feel slow at first, but it’s worth it.

To develop solid seed data for your SQL ideas, your process should be:

  1. Open your editor when your fingers start to reach for the SQL console.
  2. Record the shape and content of your tables in a way that makes them reusable during your development process.
  3. Refine the SQL scripts until they are also useful for expressing an idea to your coworkers.
  4. Polish them so they can become part of your test suite or the foundation for scripts/queries in your production codebase.

Good luck and let me know how it goes!
